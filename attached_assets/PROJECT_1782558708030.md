## 1. Présentation du projet

Plateforme SaaS verticale de réservation en ligne pour **salons de coiffure et instituts de
beauté au Maroc**, sur le modèle de Planity (France). Projet de stage ingénieur (2e année,
ENSIASDT, Maroc).

### Ce que c'est
- Un annuaire public où les clients trouvent un prestataire, consultent ses prestations/prix/
  disponibilités et réservent en ligne avec paiement intégré.
- Un outil métier SaaS où le prestataire gère son agenda, son équipe, son catalogue de
  prestations et reçoit ses réservations en temps réel.

### Ce que ce n'est pas (hors scope MVP)
- Application mobile React Native → V1 post-stage
- Intégration IA (Vercel AI SDK) → V1 post-stage
- Paiement CMI marocain natif → Stripe test mode suffit
- Multi-établissements par compte → plan Business post-MVP
- Autres verticales (médecins, restaurants…) → hors scope
- Réservation anonyme/invité → compte obligatoire

## 2. Les deux types de prestataires

Le produit supporte deux profils distincts dès le MVP :

| Type | Description | Staff |
|------|-------------|-------|
| `establishment` | Salon ou institut avec plusieurs employés | Staff multiples, chacun avec son propre agenda |
| `individual` | Auto-entrepreneur seul (coiffeur à domicile, etc.) | Un staff unique créé automatiquement à l'inscription |

## 3. Parcours utilisateurs

### Côté client (public)
1. Recherche par ville et type de prestation (sans compte)
2. Consultation de la fiche prestataire : photos, prestations, prix, équipe, avis (sans compte)
3. Sélection prestation → choix staff (ou "premier disponible") → choix créneau
4. Création de compte obligatoire (téléphone + email vérifiés) avant paiement
5. Paiement Stripe → confirmation → SMS/email de confirmation
6. Rappel automatique avant le RDV
7. Possibilité d'annuler (le créneau se libère instantanément)

### Côté prestataire (dashboard SaaS)
1. Inscription avec choix du type : `establishment` ou `individual`
2. Onboarding : infos salon, localisation, horaires d'ouverture récurrents
3. Gestion du staff : CRUD, activation/désactivation, horaires personnalisés, congés
4. Catalogue de prestations : nom, durée exacte (min), prix (centimes), buffer, staff assignés
5. Agenda en temps réel par staff (vue jour/semaine)
6. **Blocage manuel de créneaux** (walk-in) → répercution instantanée côté client via WebSocket
7. Notifications temps réel des nouvelles réservations et annulations
8. Dashboard analytics : RDV, taux de remplissage, CA estimé, avis

## 5. Modèle de données (Prisma)

### Entités principales

```
User            — clients et owners, géré par Better Auth
                  id, email (unique), phone (unique), phoneVerified,
                  name, passwordHash, role (client|owner|admin)

Provider        — un prestataire (salon ou individuel)
                  id, type (establishment|individual), name, slug (unique),
                  description, phone, email, city, latitude, longitude,
                  logoUrl, status (pending|active|suspended), ownerId FK

Staff           — membre de l'équipe d'un provider
                  id, providerId FK, name, photoUrl, bio, isActive
                  (pour un individual : auto-créé à l'inscription)

BusinessHours   — horaires récurrents par provider
                  id, providerId FK, dayOfWeek (0-6), openTime (HH:mm),
                  closeTime (HH:mm), isClosed

ScheduleBlock   — exceptions : congés, blocages manuels walk-in, absences staff
                  id, providerId FK, staffId FK (nullable),
                  type (holiday|manual_block|absence),
                  title, startDatetime, endDatetime

Service         — prestation du catalogue
                  id, providerId FK, name, description,
                  durationMinutes, priceCents, bufferMinutes, isActive

ServiceStaff    — N:N entre Service et Staff (qui peut réaliser quoi)
                  serviceId FK, staffId FK

Booking         — une réservation
                  id, providerId FK, serviceId FK, staffId FK, clientId FK,
                  startDatetime, endDatetime,
                  status (pending|confirmed|cancelled|completed|expired),
                  paymentIntentId, paymentStatus, amountCents

Review          — avis client (1 par réservation max)
                  id, providerId FK, bookingId FK (unique), clientId FK,
                  rating (1-5), comment, reply

Subscription    — abonnement SaaS du provider
                  id, providerId FK (unique), plan (free|pro|business),
                  status (active|canceled|past_due),
                  stripeSubscriptionId, currentPeriodStart, currentPeriodEnd
```

### Contrainte critique en DB
```sql
-- Filet de sécurité anti-double-booking — ne jamais supprimer
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE bookings
  ADD CONSTRAINT no_overlapping_bookings
  EXCLUDE USING GIST (
    staff_id WITH =,
    tstzrange(start_datetime, end_datetime) WITH &&
  )
  WHERE (status NOT IN ('cancelled', 'expired'));
```

## 6. Règles métier critiques

### 6.1 Anti-double-booking — défense en profondeur (3 couches)

**Couche 1 — Redis lock (première ligne)**
```
SET lock:staff:{staffId}:{slotStart} NX EX 30
```
Si le lock échoue → répondre immédiatement 409 sans toucher la DB.

**Couche 2 — Transaction PostgreSQL (deuxième ligne)**
```sql
BEGIN;
SELECT id FROM bookings
  WHERE staff_id = $staffId
  AND status NOT IN ('cancelled', 'expired')
  AND tstzrange(start_datetime, end_datetime) &&
      tstzrange($slotStart, $slotEnd)
  FOR UPDATE;
-- Si résultat non vide → ROLLBACK + 409
INSERT INTO bookings (...) VALUES (...);
COMMIT;
```

**Couche 3 — Exclusion constraint GIST (filet de sécurité)**
Voir contrainte DB ci-dessus. Ne devrait jamais se déclencher en conditions normales.

### 6.2 Slot engine — algorithme de calcul

Pour un `providerId` + `staffId` + `serviceId` + `date` donnés :
1. Charger les `BusinessHours` du jour (horaires récurrents)
2. Soustraire les `ScheduleBlocks` actifs (congés, blocages manuels, absences)
3. Charger les `Bookings` existants sur ce staff ce jour (status `confirmed` ou `pending`)
4. Générer tous les slots théoriques par pas de `durationMinutes + bufferMinutes`
5. Exclure les slots qui chevauchent une réservation ou un bloc existant
6. Retourner les slots libres avec la liste des staff disponibles par slot

Cache Redis TTL 5 min. Invalidation immédiate à chaque création/annulation/blocage.

### 6.3 Timeout paiement

- Créneau sélectionné → `status: pending`, `lockedUntil: now + 10 min`
- Job BullMQ tourne en continu → libère tous les bookings `pending` expirés
- Dès expiration : `status: expired` + émission WebSocket `slot.update` (released)
- Le client voit un compte à rebours dans l'UI

### 6.4 Synchronisation temps réel

Tout changement de disponibilité (réservation, annulation, blocage manuel, expiration)
émet immédiatement via Socket.io :
```json
{
  "type": "slot.update",
  "payload": {
    "slotStart": "09:00",
    "staffId": "uuid",
    "change": "booked|released",
    "capacityRemaining": 2
  }
}
```
Le front supprime ou réaffiche le créneau sans rechargement de page.

### 6.5 Isolation multi-tenant

- Chaque requête sur les données métier **doit** filtrer par `providerId`
- Un middleware Fastify injecte le `providerId` depuis la session (jamais depuis l'URL)
- Un prestataire ne peut jamais lire ni modifier les données d'un autre

## 7. Authentification

- **Better Auth** : sessions JWT, email/password, refresh tokens
  - Session : 24h par défaut, 7j avec "remember me"
  - Rate limiting : max 5 tentatives de connexion / 15 min
- **Firebase Auth** : vérification OTP SMS uniquement (+212 Maroc)
  - Rate limiting : max 3 envois OTP / heure / numéro
- Téléphones stockés avec indicatif +212
- La réservation est bloquée si téléphone ou email non vérifié

## 10. Exigences non fonctionnelles

- Calcul de créneaux : < 500ms pour un provider avec 10 staff sur une journée
- Cache Redis des créneaux pré-calculés sur les 7 prochains jours
- WebSocket : 100 clients simultanés par provider, sync < 2 secondes
- Disponibilité : 99.5% objectif ; Redis en panne → fallback PostgreSQL seul
- BullMQ : max 3 tentatives en cas d'échec de job
- Photos sur Cloudflare R2 (pas de stockage local)
- Timestamps en UTC, conversion fuseau côté client
