RÈGLE ANTI-GRIS-DÉGUISÉ (critique, ne pas ignorer) :
Toute valeur "ink" utilisée avec une opacité inférieure à
0.28 devient visuellement indiscernable d'un gris, peu
importe que la couleur de départ soit #0C0C0E. C'est
interdit. Cherche et corrige tout pattern de ce type :
  - style={{ backgroundColor: "var(--ink)", opacity: 0.1X }}
    avec X < 28 → remplacer par ds.colors.canvasMuted si
    c'est un fond, ou par ds.colors.inkTertiary si c'est
    du texte.
  - className="bg-ink/5" à "bg-ink/27" → interdit.
  - Tout hack du type opacity: 1 / 0.12 sur un enfant pour
    "annuler" l'opacité du parent → signe que la technique
    d'origine est fragile, remplacer par une vraie couleur
    de surface (canvasMuted) au lieu d'un noir dilué.
Plancher absolu : 0.28. En dessous, ce n'est plus "notre
noir de marque", c'est un gris qui contourne la règle.

PHASE 4 — NETTOYAGE CODE MORT

Lance ces commandes et corrige tout ce qui remonte :

1. Imports inutilisés :
   npx eslint . --rule '{"no-unused-vars": "error"}' --fix

2. Variables inutilisées :
   npx tsc --noEmit 2>&1 | grep "is declared but"

3. Fichiers jamais importés nulle part :
   find apps/web/src -name "*.tsx" -o -name "*.ts" | \
   xargs grep -L "import\|export default" | \
   grep -v "index\|main\|vite-env"

4. console.log restants :
   grep -r "console\." apps/web/src apps/api/src \
   --include="*.ts" --include="*.tsx" \
   | grep -v "console.error\|logger\."
   → Supprimer tous les console.log de debug
   → Garder uniquement console.error wrappés dans logger

5. Commentaires de code mort :
   grep -r "\/\/ TODO\|\/\/ FIXME\|\/\/ HACK\|\/\/ Phase\|\/\/ mock\|\/\/ temp" \
   apps/web/src apps/api/src \
   --include="*.ts" --include="*.tsx"
   → Pour chaque ligne : soit l'implémenter, soit la supprimer

6. Vérifier qu'il n'y a pas de fichiers dupliqués :
   find . -name "*.tsx" | sed 's/.*\///' | sort | uniq -d
   → Si des noms de fichiers apparaissent deux fois,
     vérifier lequel est utilisé et supprimer l'autre

7. Couleurs codées en dur hors du design system :
   grep -rnE "#[0-9A-Fa-f]{3,6}" --include="*.tsx" --include="*.ts" \
   apps/web/src | grep -v "design-system.ts"
   → Toute couleur hex trouvée hors de design-system.ts
     doit être remplacée par une référence ds.colors.xxx.
     Si la couleur n'a pas d'équivalent dans ds, ne pas
     l'inventer — demander l'extension de design-system.ts.

8. Noir dilué en gris déguisé :
   grep -rnE "opacity:\s*0\.(0[0-9]|1[0-9]|2[0-7])" \
   --include="*.tsx" apps/web/src
   grep -rnE "bg-ink/(0?[1-9]|1[0-9]|2[0-7])\b" \
   --include="*.tsx" apps/web/src
   → Tout résultat est une violation de la règle anti-gris-
     déguisé de la Phase 2. Corriger selon cette règle.

PHASE 5 — BONNES PRATIQUES REACT/NEXT.JS

Vérifie et corrige ces patterns dans tout le codebase :

1. Pas de fetch direct dans les composants :
   Tout appel API passe par useQuery ou dans un
   Server Component Next.js, jamais dans un useEffect
   avec fetch manuel sauf exception justifiée.

2. Gestion d'erreur sur toutes les queries :
   const { data, isLoading, error } = useQuery(...)
   if (isLoading) return <SkeletonComponent />
   if (error) return <EmptyState ... />
   Ces trois états doivent être gérés partout.

3. Typage strict :
   Toutes les réponses API doivent avoir un type défini
   dans packages/shared/src/types/
   Aucun any dans le code des composants.

4. Mutations avec feedback :
   Toute mutation (create/update/delete) doit avoir :
   onSuccess: () => toast.success('...')
   onError: () => toast.error('...')
   isPending sur le bouton : disabled + spinner visible

5. Composants atomiques :
   Si un composant dépasse 200 lignes → le découper.
   Chaque composant fait une seule chose.

6. Keys dans les listes :
   Chaque .map() utilise une key unique (jamais l'index)
   key={item.id} toujours, jamais key={index}
