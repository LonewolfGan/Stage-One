ÉTAPE 5 — SLOT ENGINE (partie critique)

Crée apps/api/src/lib/slot-engine.ts

Fonction principale :
  getAvailableSlots(providerId, staffId, serviceId, date)
  → retourne SlotResult[]

Algorithme :
  1. Charger BusinessHours du jour pour ce provider/staff
  2. Charger ScheduleBlocks actifs ce jour (congés, blocages)
  3. Charger Bookings existants status IN [PENDING, CONFIRMED]
  4. Générer tous les slots théoriques :
     de openTime à closeTime,
     par pas de (durationMinutes + bufferMinutes)
  5. Filtrer les slots qui chevauchent un booking ou un block :
     chevauchement = slotStart < existingEnd AND slotEnd > existingStart
  6. Retourner les slots libres avec staff disponibles

Mise en cache Redis :
  clé : slots:{providerId}:{staffId}:{serviceId}:{date}
  TTL : 300 secondes (5 min)
  Invalider : à chaque booking créé/annulé, à chaque block créé

Route associée :
GET /api/providers/:slug/slots?serviceId=&date=&staffId=
  - Vérifier cache Redis en premier
  - Calculer si cache miss
  - Retourner les slots disponibles

ÉTAPE 6 — RÉSERVATION AVEC ANTI-DOUBLE-BOOKING

Route :
POST /api/bookings
  Auth requise (phoneVerified ET emailVerified)
  Body : { providerSlug, serviceId, staffId?, startDatetime }

Flux exact à implémenter :

  1. Résoudre le staffId si null (premier disponible)

  2. Redis lock :
     SET lock:staff:{staffId}:{startDatetime} NX EX 30
     → Si lock échoue : retourner ERR-005 (409)

  3. Transaction PostgreSQL :
     BEGIN
     SELECT id FROM bookings
       WHERE staff_id = $staffId
       AND status NOT IN ('cancelled', 'expired')
       AND tstzrange(start_datetime, end_datetime) &&
           tstzrange($slotStart, $slotEnd)
       FOR UPDATE
     → Si résultat non vide : ROLLBACK + ERR-005
     INSERT INTO bookings (...)
     VALUES (status: PENDING, lockedUntil: now + 10min)
     COMMIT

  4. Créer PaymentIntent Stripe :
     amount: service.priceCents
     currency: 'mad'
     metadata: { bookingId }

  5. Invalider cache Redis des slots de ce staff/date

  6. Émettre via Socket.io :
     { type: 'slot.update', payload: { slotStart, staffId,
       change: 'booked' } }

  7. Retourner : { bookingId, status: 'pending',
     paymentIntentSecret, expiresAt }

Job BullMQ — expiration paiement :
  Toutes les minutes, libérer les bookings PENDING
  dont lockedUntil < now() :
  UPDATE bookings SET status = 'EXPIRED'
  WHERE status = 'PENDING' AND locked_until < now()
  Émettre slot.update (released) pour chaque libéré
  Invalider cache Redis correspondant

RÈGLES ABSOLUES

1. Ne jamais modifier un composant React ou un fichier CSS existant
2. Ne jamais retourner le providerId depuis l'URL — toujours
   depuis la session JWT
3. Toujours valider les entrées avec Zod avant traitement
4. Toujours invalider le cache Redis après toute modification
   d'un booking, d'un block ou d'un service
5. La contrainte GIST doit être dans la migration initiale —
   ne jamais l'omettre
6. Tester explicitement le cas de deux requêtes simultanées
   sur le même créneau avant de valider le lot 4
