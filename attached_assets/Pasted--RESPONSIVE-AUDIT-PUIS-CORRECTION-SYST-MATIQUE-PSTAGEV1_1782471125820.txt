# RESPONSIVE — AUDIT PUIS CORRECTION SYSTÉMATIQUE, PSTAGEV1

> Ce prompt traite UNIQUEMENT le responsive et l'intégrité structurelle du
> layout — pas la couleur, pas la typographie, pas les radius. Le design
> system (`tailwind.config.ts`) reste la seule source de vérité pour les
> valeurs visuelles et ne change pas. Ce qu'on corrige ici : que chaque
> page, chaque section, chaque composant (front public ET dashboard admin)
> reste fonctionnel et cohérent à toutes les tailles d'écran, sans
> exception, et selon une philosophie de breakpoints unique — pas des
> correctifs isolés ajoutés au cas par cas.

Ne fais rien d'autre que ce qui est décrit ici. N'ajoute pas de nouvelle
fonctionnalité, ne change pas de couleurs ou de composants visuellement —
uniquement la structure responsive.

### 1.2 Détection mécanique des patterns à risque

Exécute ces commandes et inclus leur sortie brute dans le rapport :

```bash

# Largeurs ou hauteurs fixes en pixels qui ne s'adaptent pas

# Texte en taille fixe sans variante mobile (à croiser avec le design system :
# les classes sémantiques type text-display-hero doivent avoir un équivalent

# Sidebar ou nav fixes en largeur sans comportement mobile (drawer/collapse)

# Tableaux/grilles dashboard qui débordent souvent sur mobile
```

## PHASE 2 — PHILOSOPHIE RESPONSIVE UNIQUE (à adopter avant de corriger)

Le but n'est pas d'ajouter des `md:` un peu partout au hasard, mais
d'appliquer un **système de breakpoints identique sur toute la
plateforme**, public et admin confondus.

### 2.1 Breakpoints — les seuls à utiliser, partout

| Nom Tailwind | Largeur | Usage |
|---|---|---|
| (défaut, pas de préfixe) | 0–639px | Mobile. Layout TOUJOURS en une seule colonne empilée verticalement. |
| `sm:` | 640px+ | Mobile large. Ajustements mineurs seulement (padding, taille de texte). |
| `md:` | 768px+ | Tablette. Premier point où une grille à 2 colonnes ou un layout sidebar+contenu peut apparaître. |
| `lg:` | 1024px+ | Layout desktop complet : grilles à 3+ colonnes, sidebar fixe visible, hero asymétrique pleinement déployé. |
| `xl:` | 1280px+ | Affinages de densité sur grand écran uniquement (jamais de changement structurel majeur ici). |

**Règle absolue** : le comportement mobile (`< 640px`) n'est jamais un
"à peu près" — c'est une vraie colonne unique, lisible, sans aucun élément
qui dépasse horizontalement. Si un composant n'a pas explicitement de
classes `sm:`/`md:`/`lg:` qui changent sa disposition, c'est qu'il doit
fonctionner correctement EN L'ÉTAT à 375px — vérifie-le, ne suppose pas.

### 2.2 Règles structurelles obligatoires, par type de composant

**Grilles de cards (résultats recherche, catégories, services)**
```
grid-cols-1 sm:grid-cols-2 lg:grid-cols-3
```
Jamais de `grid-cols-3` ou plus sans les variantes mobile/tablette qui
précèdent. Le gap réduit aussi : `gap-4 lg:gap-6` plutôt qu'un gap fixe.

**Hero et sections avec titre + élément secondaire**
Le titre (`text-display-hero`) DOIT avoir un équivalent mobile dans le même
élément :
```
text-display-hero-mobile md:text-display-hero
```
Toute composition asymétrique (texte à gauche + élément à droite) s'empile
verticalement sous `lg:` :
```
flex flex-col lg:flex-row lg:items-start
```

**Navigation publique (TopBar)**
Sous `lg:`, les liens de navigation centraux disparaissent et sont
remplacés par un menu burger/drawer — jamais compressés en police minuscule
ni laissés à déborder.
```
hidden lg:flex   {/* nav desktop */}
flex lg:hidden   {/* bouton burger mobile */}
```

**Dashboard — Sidebar**
Sous `lg:`, la sidebar fixe de 220-224px ne doit JAMAIS rester affichée en
permanence en écrasant le contenu. Elle devient soit un drawer qui s'ouvre
par-dessus (overlay), soit une bottom-nav, soit un menu burger qui l'ouvre
en plein écran temporairement :
```
hidden lg:flex lg:w-[224px]   {/* sidebar desktop fixe */}
```
Le contenu principal du dashboard ne doit jamais être compressé à côté
d'une sidebar sur un écran < 1024px.

**Tableaux et grilles denses (dashboard agenda, listes de réservations)**
Sous `md:`, un `<table>` classique ne doit jamais déborder horizontalement
sans solution : soit transformation en liste de cards empilées sous `md:`,
soit `overflow-x-auto` explicite AVEC une indication visuelle qu'il y a du
contenu caché (pas juste un scroll silencieux).

**Formulaires et inputs (booking, onboarding, paramètres)**
Toujours `w-full` par défaut, jamais de largeur fixe en `px`. Les champs
côte à côte sur desktop (`flex gap-4`) s'empilent verticalement sur mobile
(`flex-col sm:flex-row`).

**Images (galerie fiche prestataire, photos profil staff)**
Toujours contraintes par leur container (`w-full h-full object-cover`),
jamais de largeur fixe en pixels qui pourrait dépasser l'écran.

### 2.3 Interdictions permanentes, vérifiables par grep

```bash
# Largeur fixe en pixels sur un élément de layout (hors icônes/avatars < 64px)
grep -rEn "\bw-\[[0-9]{3,}px\]" --include="*.tsx" apps/web/app apps/web/components

# Grille fixe sans variante responsive
grep -rn "grid-cols-[3-9]\b" --include="*.tsx" apps/web/app apps/web/components | grep -v "sm:\|md:\|lg:"

# Sidebar toujours visible sans variante hidden/lg:flex
grep -rn "w-\[224px\]\|w-\[220px\]" --include="*.tsx" apps/web/app apps/web/components | grep -v "hidden lg:\|lg:flex"

# overflow-x sans indication visuelle (scroll silencieux sur tableau)
grep -rln "overflow-x-auto" --include="*.tsx" apps/web/app apps/web/components
```

Ces commandes doivent toutes retourner soit 0 résultat, soit uniquement des
résultats déjà accompagnés d'une variante responsive justifiée — vérifie
chaque match individuellement, ne te contente pas du compte.

## PHASE 3 — CORRECTION

Une fois le rapport de la phase 1 produit et la philosophie de la phase 2
acceptée, corrige fichier par fichier, dans cet ordre :

1. **Layout racine** (`app/layout.tsx`) et TopBar/Footer publics — ce sont
   les éléments présents sur toutes les pages, donc les corriger en premier
   évite de répéter le même bug partout.
2. **Sidebar dashboard et layout dashboard** (`app/dashboard/layout.tsx` ou
   équivalent) — même logique, présent sur toutes les pages admin.
3. **Hero et page d'accueil.**
4. **Page de recherche/résultats** (grilles de cards).
5. **Fiche prestataire** (galerie + services + booking widget — c'est
   probablement la page la plus à risque, elle combine image, grille et
   formulaire).
6. **Pages dashboard restantes** (agenda, prestations, staff, stats).
7. **Formulaires** (auth, onboarding, paramètres) en dernier — ce sont
   souvent les plus simples une fois le reste stabilisé.

Pour chaque fichier corrigé, refais le test des 5 largeurs (section 1.3) et
confirme avec une preuve concrète (pas une affirmation verbale) que les 5
largeurs fonctionnent avant de passer au fichier suivant.

À la fin de la phase 3, relance l'intégralité des commandes grep de la
section 2.3 sur tout `apps/web` et montre la sortie complète. Le travail
n'est terminé que lorsque ces commandes ne révèlent plus aucun problème non
justifié.
